The Sidecar Pattern

Sometimes our main application needs a helper. A "Sidecar" is a secondary container deployed right next to our primary application container, exclusively there to provide supporting features like logging, metrics, or a reverse proxy.

services:
  # The Primary App
  api:
    image: my-python-app
    ports:
      - "8000:8000"

  # The Sidecar (Datadog Agent gathering metrics)
  datadog:
    image: datadog/agent:latest
    environment:
      - DD_API_KEY=${DD_API_KEY}
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock:ro

By splitting them, if the Datadog agent crashes, our Python API stays perfectly alive. We decouple our monitoring from our business logic.

Init Containers

Sometimes, we must guarantee that a specific task finishes before our main application boots up. For example, running strict configuration generation, or waiting for an external service.

Init Pattern Explained:
We run a temporary container that does the work. Our main application uses depends_on: [init_container] with a condition: service_completed_successfully. If the Init container fails, the main app never boots!

One-off Task Containers

We often need to run a temporary script—like downloading a specific file, parsing a JSON, or generating a quick RSA key—but we absolutely do not want to pollute our laptop with global node modules or curl configurations.

docker run --rm alpine sed...
The --rm flag is magical. It tells Docker to boot the container, run the single command, and the exact millisecond the command finishes, delete the container permanently. 
# Generate a quick secure secret securely without installing openSSL locally
docker run --rm alpine sh -c "apk add openssl && openssl rand -hex 32"

Database Migration Containers

How do we run our schema migrations (like Django Alembic, Prisma, or TypeORM) in production? We do NOT put the `migrate` command directly inside our Web API's CMD! If we do, when we scale up to 10 web servers, 10 servers will all violently try to migrate the exact same database simultaneously, causing horrific lock corruption.

Instead, we run one dedicated migration container during deployment before we launch the new web servers.

services:
  # The one-off Migration Runner
  migrator:
    image: my-app:latest
    command: npm run prisma migrate deploy # Runs the migration then exits!
    environment:
      DATABASE_URL: ${DB_URL}

  api:
    image: my-app:latest
    depends_on:
      migrator:
        condition: service_completed_successfully # The API waits for the green light
    command: npm run start

Cron Job Containers

If we want to run a script every 5 minutes (like cleaning up old sessions or scraping an API), we don't need a heavy scheduling framework. We can literally run a container whose sole purpose is sleeping and running a loop. 

# Example: A container that hits an endpoint every 60 seconds forever
services:
  ping_bot:
    image: alpine:latest
    command: sh -c "while true; do wget -qO- http://api:8000/health; sleep 60; done"

Health Checks in Production

Docker naturally assumes a container is "Healthy" as long as the process hasn't crashed. But what if our Node app is stuck in a deadlocked While Loop? The process is running, but our users see infinite timeouts. We must use a HEALTHCHECK.

FROM python:3.11-slim
# ... setup app ...

# Every 30 seconds, Docker will literally ping our internal endpoint. 
# If it fails 3 times in a row, Docker marks the container as "Unhealthy".
HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
  CMD curl -f http://127.0.0.1:8000/health || exit 1

CMD ["uvicorn", "main:app"]
⚠️ Healthcheck and Load Balancers:
Docker Compose doesn't automatically restart Unhealthy containers by default. However, advanced orchestrators like Docker Swarm or Kubernetes explicitly rely on this specific script to automatically kill and replace frozen containers. Never skip it in production.

Container Auto-Restart Policies

If our Node server suddenly crashes due to a massive memory spike, what happens? By default, the container dies and stays completely dead forever. In production, we need a guarantee of high availability.

We use the --restart flag, or the restart: directive in Compose.

Policy How It Operates
no (Default) If it crashes, it's done. Never restarts automatically.
always If it crashes, Docker aggressively restarts it. If we restart the host physical server, Docker instantly reboots the container on startup. Only way to permanently stop it is by explicitly running docker stop.
unless-stopped (The Production Standard) Acts identical to `always`, EXCEPT if we explicitly typed `docker stop` before the server rebooted, Docker respects our wishes and leaves it offline.
on-failure Only restarts if the internal script crashed throwing a tragic error code (Not `0`). Leaves it alone if the script finished its job peacefully.

Logging Drivers Overview

By default, Docker captures all standard output into giant JSON files heavily hidden on the host OS. If our app logs millions of events, these files can rapidly consume our entire hard drive.

We solve this by explicitly limiting log size, or completely shipping our logs directly to external logging tools (like Datadog, AWS CloudWatch, or Splunk) via Logging Drivers.

services:
  api:
    image: my-app:latest
    logging:
      driver: "json-file" # Ensure we use the default driver
      options:
        max-size: "10m"   # Maximum 10MB per log file
        max-file: "3"     # Keep only the 3 most recent files, then overwrite!
