Docker Desktop vs. Docker Engine

Before installing, we must understand the difference between the two main versions of Docker:

  • Docker Engine: The raw, headless background process that runs containers. It only runs natively on Linux servers.
  • Docker Desktop: A graphical application built for Mac and Windows. Because containers explicitly require a Linux kernel, Docker Desktop secretly spins up a lightweight Linux virtual machine on our computer, installs Docker Engine inside it, and bridges it to our host OS so we can run Linux containers on a Mac or Windows machine.
Rule of Thumb:
If we are on a Mac or Windows laptop, we install Docker Desktop.
If we are deploying code to a production Ubuntu or Debian server, we install the raw Docker Engine.

Installation Guides

1. Windows (WSL2 + Docker Desktop)

Windows developers must use the Windows Subsystem for Linux (WSL2), which runs a true Linux kernel alongside Windows. Docker Desktop integrates directly with WSL2.

  1. We open PowerShell as Administrator and run: wsl --install. Then restart our computer.
  2. Download and install Docker Desktop for Windows from the official website.
  3. During installation, ensure the checkbox for "Use WSL 2 instead of Hyper-V" is checked.
  4. Open Docker Desktop and wait for the engine to start (the whale icon bottom-left should turn green).
💡 WSL2 Integration Tip:
To get maximum file I/O performance on Windows, we keep our project code files inside the WSL2 filesystem (e.g., \\wsl$\Ubuntu\home\myproject) rather than our standard Windows `C:\` drive. If Docker has to constantly translate files back-and-forth across the Windows/Linux boundary, our container builds will be incredibly slow.

2. macOS

MacOS relies on a hidden hypervisor to run Linux containers. The installation is perfectly streamlined.

  1. Download Docker Desktop for Mac (making sure to pick the correct Apple Silicon vs. Intel chip version).
  2. Drag the app to Applications and launch it.
  3. Accept the permissions to install the background networking tools.

3. Linux (Ubuntu Server)

On Linux, we do not install Docker Desktop. We want the bare-metal Docker Engine for maximum performance. We use the official repository:

# 1. Update our apt package index
sudo apt-get update

# 2. Install certificates and curl
sudo apt-get install ca-certificates curl

# 3. Add Docker's official GPG key
sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc

# 4. Add the repository to Apt sources:
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] https://download.docker.com/linux/ubuntu \
  $(. /etc/os-release && echo "$VERSION_CODENAME") stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# 5. Install the Engine
sudo apt-get update
sudo apt-get install docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin

Verifying the Installation

Once installed, we open a fresh terminal tab and run these two absolute checks:

docker version

This command splits output into two halves: "Client" and "Server". If we see both blocks printed out, we are good to go. If it says "Cannot connect to the Docker daemon", it means our background engine has not fully booted up. 

docker info

This dumps the total system state, confirming our exact architecture (e.g., `aarch64` for Apple Silicon) and how many containers and images exist.

Docker Hub Account & Login

To pull specialized images or upload our own code later, we need a free Docker Hub account.

  1. Go to hub.docker.com and create a free account.
  2. Open our terminal and type docker login.
  3. Enter our username (not email) and password or Personal Access Token (PAT).

Our local Docker daemon is now cryptographically linked to our cloud registry.
