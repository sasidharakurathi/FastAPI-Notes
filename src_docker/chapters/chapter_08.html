The Default Networks

When we install Docker, it automatically creates three default virtual networks. We can see them by running docker network ls.

Network Type How It Works
bridge (Default) A private internal walled-garden. Containers placed here can talk to the internet, but the outside world cannot reach them unless we explicitly drill a hole (publish a port).
host The container completely abandons isolation and shares the exact IP address and networking stack of our host laptop/server.
none Total isolation. The container has no access to the internet or any other containers. (Extremely rare, used for high-security offline processing).

Network Management Commands

By default, if we don't specify a network, every new container gets dumped into the default `bridge`. But best practice dictates we create a custom network for every application layout.

docker network create my_app_net
Creates a brand new virtual bridge network.

docker network inspect my_app_net
Shows a JSON dump of exactly which containers are currently connected to this network and what internal IP addresses they have been assigned.

docker network connect my_app_net my_container
Attaches an already-running container into our newly created network dynamically.

Container-to-Container Communication (Internal DNS)

If we want our Python backend to talk to our PostgreSQL database, we could try logging into the database container, finding its internal IP (e.g., `172.17.0.4`), and hardcoding that into our Python app.

This is a terrible idea. Internal IPs change every time a container restarts.

Instead, Docker provides a magical internal DNS Resolver. If we put two containers on the same custom network, Docker allows them to talk to each other using their container names as pure hostnames.

Real Example: Python Talking to Postgres

Let's create a custom network and attach two containers so they can talk securely behind the scenes.

# 1. Create a private network
docker network create backend_net

# 2. Start Postgres, attach it to our network, and name it 'db'
docker run -d --name db --network backend_net \
  -e POSTGRES_PASSWORD=secret postgres:15

# 3. Start Python, attach to the SAME network.
docker run -it --name api --network backend_net python:3.11-slim bash

Now, from inside the `api` container's bash prompt, if our Python code tries to connect to the database, our Database URL doesn't use `localhost` or an IP address. It uses the exact container name:

# Inside the Python container
DATABASE_URL="postgresql://postgres:secret@db:5432/postgres"

Docker's internal DNS dynamically intercepts the word db and perfectly routes the traffic to the Postgres container!

Publishing vs. Internal Networking (-p Flags)

One of the biggest mistakes beginners make is thinking they must use -p 5432:5432 on their database so their backend container can talk to it.

⚠️ Warning: Unnecessary Port Publishing
Port publishing (-p) drills a hole from the outside host machine (our laptop) into the container. Containers on the same Docker network can ALWAYS talk to each other on ANY port internally. If our web app talks to our database, we do not need to publish the database port. By NOT publishing it, we keep the database 100% hidden and secure from the outside world.

We only use -p 8080:80 when we explicitly want our laptop's web browser, or the public internet, to reach inside the container.

Host Networking: Linux vs. Mac/Windows

Sometimes we want maximum performance, so we use --network host. This removes all networking virtualization overhead.

Platform How `host` Networking Behaves
Ubuntu / Linux Servers Works perfectly. The container directly uses the server's network interfaces with zero latency.
Mac & Windows (Docker Desktop) Silently fails. Because Docker Desktop actually runs inside a hidden Linux VM, attaching to the "host" network attaches it to the hidden VM's network, NOT our actual Mac/Windows laptop.

If we are developing on a Mac or Windows machine, we must rely on -p port forwarding. We save --network host strictly for production Linux deployments.
