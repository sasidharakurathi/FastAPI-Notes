CHAPTER 5 — Writing Dockerfiles (The Core Skill)

The Build Context and .dockerignore

Before we even write a Dockerfile, we must understand the Build Context. When we type docker build ., the Docker CLI gathers every single file and folder in the current directory and sends it to the Docker Daemon to be processed.

⚠️ The 1GB Upload Trap:
If we don't filter our context, Docker will upload our massive node_modules/, our virtual environments, and our `.git/` folder to the daemon on every single build. This makes builds incredibly slow and bloated.

To fix this, we ALWAYS create a .dockerignore file in the root of our project. It works exactly like a `.gitignore`.

# .dockerignore
.git
node_modules/
venv/
__pycache__/
.env

The Core Instructions

A Dockerfile is a simple script. Let's break down the actual commands we use to construct our image layers.

Command What it does
FROM Must be the first line. Sets the base image (e.g., FROM python:3.11-slim).
WORKDIR Creates a folder inside the container and CD's into it. All following commands run here (e.g., WORKDIR /app).
COPY Copies files from our laptop into the container (e.g., COPY package.json .).
ADD Like COPY, but it can also auto-extract `.tar.gz` files and download from URLs. (We generally prefer COPY unless we specifically need ADD's magic).
RUN Executes a terminal command during the build phase to install things (e.g., RUN pip install -r requirements.txt). Creates a new layer.

More Configuration Instructions

Command What it does
ENV Sets persistent environment variables that exist while the container is running (e.g., ENV NODE_ENV=production).
ARG Like ENV, but only exists during the build phase. Useful for passing in dynamic build versions.
EXPOSE Purely documentation. Tells other developers which port this container listens on (e.g., EXPOSE 8080). Does NOT actually publish the port.
USER Switches from the default `root` user to a safer, restricted user for security (e.g., USER node).
VOLUME Marks a specific directory to be mounted as a permanent volume, bypassing the ephemeral filesystem.
LABEL Adds metadata to an image (e.g., LABEL maintainer="team@example.com").
HEALTHCHECK Tells Docker how to test if our app is actually healthy, not just if the process is running.
ONBUILD A trigger instruction that executes only if someone else uses OUR image as their base FROM image.

CMD vs. ENTRYPOINT (The Ultimate Confusion)

Both of these commands dictate what our container actually DOES when it finishes booting. But they have a crucial difference.

CMD (The Default Suggestion)
CMD ["python", "app.py"]
This provides a default command. However, if a user types docker run my-app bash, the user's `bash` command completely overrides and replaces our CMD.

ENTRYPOINT (The Hard Rule)
ENTRYPOINT ["python", "app.py"]
This locks the container into running this exact executable. If a user types docker run my-app --help, Docker appends `--help` as an argument to our Entrypoint, effectively running `python app.py --help`.

Best Practices: Layer Caching & Ordering

Docker caches layers to speed up builds. If a layer hasn't changed, Docker instantly reuses the cached version. However, if one layer changes, every single layer below it is forcibly rebuilt.

Because source code changes frequently, but dependencies (like libraries) change rarely, we must ALWAYS copy our dependency files and install them before we copy our source code.

Example 1: Dockerizing a FastAPI (Python) App

FROM python:3.11-slim

# Create our working directory
WORKDIR /app

# COPY ONLY the tiny requirements file first!
COPY requirements.txt .

# Install dependencies (This layer gets heavily cached!)
RUN pip install --no-cache-dir -r requirements.txt

# NOW copy our frequently-changing source code
COPY . .

# Run as a non-root user for security
RUN useradd -m myuser
USER myuser

EXPOSE 8000

CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

Example 2: Dockerizing a Node.js/Express App

The exact same caching logic applies to Node.js applications.

FROM node:20-alpine

WORKDIR /app

# Copy only the package files first
COPY package.json package-lock.json ./

# Install npm dependencies securely
RUN npm ci --only=production

# Copy the rest of the application
COPY . .

# Switch to the pre-existing 'node' user in the alpine image
USER node

EXPOSE 3000

CMD ["node", "server.js"]

Example 3: Dockerizing a React App (Multi-Stage Build)

A React app requires heavy compilation tools (Node, Webpack) to build the static HTML/JS, but to actually serve those files, we only need a tiny, lightweight Nginx server. Multi-stage builds let us use multiple `FROM` statements to leave the heavy build tools behind.

# --- Stage 1: The Builder ---
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build
# The compiled files now live in /app/dist

# --- Stage 2: The Production Server ---
FROM nginx:alpine
# We ONLY copy the final built files from Stage 1. 
# Node.js and all its heavy modules are completely discarded!
COPY --from=builder /app/dist /usr/share/nginx/html/

EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

This multi-stage approach reduces our final image size from ~1GB down to a shockingly lean ~20MB!
