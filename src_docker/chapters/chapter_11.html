CHAPTER 11 — Docker Registry & Image Publishing

The Cloud Storage Mental Model

Writing a Dockerfile and building an image locally is fantastic, but we eventually need to put that image on a production server. We don't zip up the image and email it. We use a Registry.

A Registry is effectively an "App Store" or "GitHub" strictly for Docker images.

  • Public Repositories: Anyone in the world can docker pull our image without logging in. Perfect for open-source tools (like Nginx or Ubuntu).
  • Private Repositories: Requires explicit authentication via docker login with a password or Personal Access Token (PAT). Essential for proprietary company code.

The Image Naming Convention

Before we can push anything to the cloud, our image must follow a strict structural naming convention:

registry_domain/username/repository_name:target_tag
Breaking down: ghcr.io/microsoft/devcontainers:latest

ghcr.io: The server hosting the registry (GitHub). If missing, Docker assumes `docker.io` (Docker Hub) by default.
microsoft: The user or organization account that owns the repository.
devcontainers: The specific application repository.
:latest: The specific version tag.

Real Example: Pushing to Docker Hub

Let's walk through the exact steps a developer takes to publish an image to the default Docker Hub.

  1. Authenticate: We must prove to the daemon who we are. 
    docker login
# It will prompt for our Docker Hub username and password/PAT.
  2. Build the Image: We build the code locally, assigning it a temporary name. 
    docker build -t my-app .
  3. Retag the Image: We MUST tag the image to match our exact Docker Hub username, otherwise the upload will be rejected for lack of permission. Let's assume our username is `cooldev`. 
    # docker tag [SOURCE_IMAGE] [TARGET_NAME]
docker tag my-app cooldev/my-app:v1.0
  4. Push the Image: We upload the newly tagged blueprint to the cloud. 
    docker push cooldev/my-app:v1.0

Our production server can now easily download this image across the world by running docker pull cooldev/my-app:v1.0.

GitHub Container Registry (ghcr.io)

Docker Hub is perfectly fine, but many teams already have their code sitting in GitHub. GitHub offers its own highly-integrated registry named GHCR.

# 1. Create a Personal Access Token (PAT) on GitHub
# It MUST have 'read:packages' and 'write:packages' scopes.

# 2. Login to the specific GitHub registry domain
echo $MY_GITHUB_PAT | docker login ghcr.io -u my_github_username --password-stdin

# 3. Tag exactly by prepending the domain
docker tag my-app ghcr.io/my_github_org/my-app:v1.0

# 4. Push to GitHub
docker push ghcr.io/my_github_org/my-app:v1.0

Hosting Our Own Private Registry

If our company has extreme "air-gapped" security requirements where code cannot touch the internet, we can actually spin up an entire Docker Registry locally using Docker itself.

# Run the official, isolated local registry on port 5000
docker run -d -p 5000:5000 --name local-registry registry:2

We can now tag and push deeply sensitive proprietary images directly to localhost:5000/top_secret_app:v1.

Automating Builds (GitHub Actions)

We never want a human manually typing docker push from their laptop in production. Laptops have dirty states and uncommitted files. A true Senior Developer offloads this entirely to a CI/CD pipeline.

When we push code to the `main` branch, we want GitHub Actions to automatically securely build and push our final production image to GHCR.

# .github/workflows/docker-publish.yml
name: Build and Publish

on:
  push:
    branches: [ "main" ] # Trigger specifically on main

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write

    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Login to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ghcr.io/${{ github.repository }}:latest
⚠️ CI/CD Security Tip:
Notice that we never hardcoded passwords in that YAML file. The ${{ secrets.GITHUB_TOKEN }} is an ephemeral, auto-generated token created by GitHub specifically for that single workflow run. It is the absolute safest way to authenticate our automated builders.
