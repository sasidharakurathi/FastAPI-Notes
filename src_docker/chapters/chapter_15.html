Essential Docker CLI Commands

Command What it does
docker run -d -p 80:80 my-app Starts a container detached, publishing port 80 to the host.
docker ps / docker ps -a Lists all running containers / Lists every container (even stopped ones).
docker stop <id> / rm <id> Gracefully shuts down a container / Permanently deletes a stopped container.
docker exec -it <id> bash Opens an interactive bash terminal directly inside a running container.
docker logs -f <id> Streams the container's standard output (console logs) in real-time.
docker build -t my-app . Builds an image from the Dockerfile in the current directory and tags it.
docker images / rmi <name> Lists all locally cached images / Deletes a specific image from our hard drive.
docker network ls Lists all virtual networks bridging our containers together.

Dockerfile Instruction Cheat Sheet

Instruction Description
FROM Defines the base image (e.g., FROM python:3.11-slim). Must be first.
WORKDIR Creates and enters a directory. All subsequent commands run from here.
COPY / ADD Copies files from our laptop into the image. (`COPY` is preferred over `ADD`).
RUN Executes a bash command during the build phase (e.g., `npm install`).
ENV / ARG Sets environment variables. (`ENV` persists at runtime, `ARG` is only for building).
USER Switches the executing user away from default `root` for security.
CMD / ENTRYPOINT The final command that executes when the container actually boots up.

Docker Compose Cheat Sheet

docker compose up -d --build
Forces a fresh rebuild of our Dockerfiles, then starts the entire network in the background.

docker compose down -v
Tears down all containers and networks, and uses -v to securely wipe all attached nameless volumes.

docker compose logs -f api
Follows the real-time logs specifically for the service named `api`.

Debugging Common Errors

1. "bind: address already in use"

The Error: Something on our laptop is already using the port we tried to publish via -p 8080:8000.

The Fix: We either stop the rogue local service (maybe another container is still running), or simply change our host port parameter to -p 8081:8000.

2. "no space left on device"

The Error: Docker has completely filled our laptop's hard drive with unused gigabyte-heavy images and dead containers.

The Fix: We must aggressively prune the Docker Daemon's cache. (See next page for the ultimate cleanup command).

3. "permission denied (bind mount)"

The Error: Our container is trying to write to a bind-mounted folder (e.g., logging to `./logs`), but the Host OS (Mac/Linux) is blocking the container's internal non-root user.

The Fix: We make sure the folder on our laptop has appropriate wide-open write permissions (chmod 777 ./logs for local dev), or we configure User Namespace Remapping.

4. "image my-app:latest not found"

The Error: We told Docker Compose to boot `image: my-app`, but we haven't actually built it yet, and Docker Hub definitely doesn't host our private app.

The Fix: We must explicitly run docker compose build, OR we must add a build: . block underneath the `image:` block so Compose knows how to build it automatically.

5. "network not found"

The Error: We ran `docker run --network my_net`, but Docker throws a fatal error because the custom network doesn't physically exist.

The Fix: When running independent containers, we must manually run docker network create my_net first. (Remember, Docker Compose handles this automatically, but plain `docker run` does not).

The Ultimate System Cleanup

Over months of development, Docker endlessly hoards old images, dangling build caches, stopped containers, and orphaned volumes. This can easily consume 50GB to 100GB of our laptop's SSD.

When our system is bloated or corrupted, we drop the nuclear bomb to completely reset our environment to a pristine state:

docker system prune --all --volumes

Warning: This permanently deletes ALL stopped containers, ALL unused virtual networks, ALL unattached images, and ALL dangling volumes. It is completely unrecoverable. But it almost always fixes the hardest hidden caching bugs.
