CHAPTER 12 - Docker in CI/CD

Why Docker in CI? (Reproducibility)

Continuous Integration (CI) servers execute our automated tests. If our tests rely on the CI server having exactly Python 3.11 and Postgres 15 installed globally, our tests will eventually break. A pipeline that passes today might fail tomorrow just because the CI runner auto-updated its global Python version.

The Goal:
We want 100% Hermetic Reproducibility. We package everything inside a Docker container, run our tests inside that exact container, and if it passes, we push that exact identical container to production. We test the exact identical artifact we plan to ship.

The Common Pattern

A Senior Developer's CI/CD pipeline looks like exactly this on every Pull Request:

  1. Build: Construct the Docker Image dynamically in the CI runner.
  2. Test: Run our unit tests entirely inside the newly built ephemeral container.
  3. Push: If (and only if) the PR is safely merged to `main`, tag the image as `latest` and push it to our Cloud Registry.

GitHub Actions: Caching Layers

GitHub Actions runners boot up completely fresh and naked every single time. This means we instantly lose all of our powerful Docker Layer Caching (which makes builds fast). If we don't fix this, a 5-second build on our laptop will brutally take 5-10 minutes in CI.

We fix this by using the `cache-from` and `cache-to` flags, completely offloading our Docker layer cache directly to GitHub's internal fast-cache storage API.

# .github/workflows/ci.yml
steps:
  - name: Set up Docker Buildx
    uses: docker/setup-buildx-action@v3

  - name: Build and Export Cache
    uses: docker/build-push-action@v5
    with:
      context: .
      # Don't push yet! Wait for tests to pass.
      push: false
      load: true 
      tags: my-app:test
      # Automatically pull/push layers to GitHub's Cache API
      cache-from: type=gha
      cache-to: type=gha,mode=max

  - name: Run Tests Inside Container
    run: |
      docker run --rm my-app:test pytest /app/tests

GitLab CI: DinD vs Socket Mounting

If we use GitLab CI, the CI runner itself is natively executing inside a Docker container. This creates a deeply confusing "Inception" scenario: How do we run Docker commands inside a Docker container?

Strategy How It Works Pros / Cons
Socket Mounting We inject the Host server's literal /var/run/docker.sock file directly inside the CI container as a Bind Mount. Pro: Blazing fast. Best for layer caching.
Con: Severe security risk. The CI container gets root access to the host server.
Docker-in-Docker (DinD) We run an entire isolated, nested Docker Daemon runtime literally inside the CI container. (The docker:dind secondary service). Pro: Highly secure isolation.
Con: Slower. Destroys layer caching because the nested daemon shuts down permanently after the pipeline finishes.		 
# .gitlab-ci.yml (DinD Example)
test_job:
  image: docker:24.0.5
  services:
    - docker:24.0.5-dind
  variables:
    DOCKER_HOST: tcp://docker:2375
  script:
    - docker build -t my-app .
    - docker run --rm my-app npm test

Integration Testing with Compose

Unit tests are great, but what if our code requires an actual working PostgreSQL database and Redis cluster to pass its tests? We literally spin up our entire docker-compose.yml inside the CI pipeline.

# Inside our CI workflow bash script

# 1. Boot up the entire architecture in the background
$ docker compose up -d

# 2. Wait exactly 5 seconds for the database to boot and accept connections
$ sleep 5

# 3. Aggressively execute our integration tests inside the running API container
$ docker compose exec -T api pytest /app/tests/integration

# 4. Tear the entire infrastructure down and destroy the ephemeral database
$ docker compose down -v
⚠️ CI TTY Error:
Notice we used exec -T in the script instead of our usual exec -it. CI servers do not have physical keyboards or interactive terminal drivers. Using -it in a GitHub Action will crash immediately with a "the input device is not a TTY" error.
