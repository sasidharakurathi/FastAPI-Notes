What Exactly is an Image? (The Layer Cake)

We often call a Docker Image a "blueprint". But structurally, an image is built like a stack of transparent pancakes, called Layers.

Every single instruction (like RUN apt-get install python) in our Dockerfile creates a new, permanent, read-only layer built on top of the previous one. When Docker pulls an image from the cloud, it doesn't download one giant file; it smartly downloads these individual layers.

Why layers are brilliant:
If we have 10 different apps that all use ubuntu:latest as their base, Docker doesn't download Ubuntu 10 times. It securely shares the single cached Ubuntu layer across all 10 containers on our hard drive. This saves massive amounts of disk space and download time.

Finding & Managing Images

Let's look at the core commands we use to interact with our images locally:

docker images
Lists all the images currently cached on our machine, showing their size and tags.

docker pull <image_name>
Force-downloads an image from Docker Hub (or another registry) to our laptop without running it.

docker rmi <image_id>
"Remove Image". Deletes the image from our hard drive to free up space. (Note: We cannot delete an image if a container-even a stopped one-is currently currently attached to it).

Diving Deep: Docker Inspect

Sometimes an image behaves strangely, and we need to know exactly what is inside it without spinning it up.

docker inspect nginx

This command returns a massive JSON file containing the absolute truth about the image. We use this to find out:

  • Exactly which ports it intends to expose (the `ExposedPorts` field).
  • Any default Environment Variables it expects us to pass.
  • The default `Cmd` or `Entrypoint` (what command it automatically runs on boot).

Official vs. Custom Images

When searching Docker Hub, we see millions of images. We must be highly selective about which ones we trust to run on our servers.

  • Official Images: Handled by Canonical (Ubuntu), Docker, or the specific open-source foundation (e.g., Python, Node, Redis). They are actively scanned for security vulnerabilities. We should heavily favor these as our bases.
  • Verified Publisher Images: Built by trusted companies like Bitnami. Generally very safe, but we must verify the publisher checkmark on Docker Hub.
  • Random Community Images: E.g., `someguy99/cool-nginx-hack`. Never use these in production. They might contain outdated software, massive security holes, or literal crypto-miners.

The Tagging Strategy

Every image has a name (like postgres) and a tag (like :15-alpine). A tag acts exactly like a version number.

⚠️ The "Latest" Trap:
If we don't specify a tag, Docker automatically applies the :latest tag. Beginners love using node:latest. Do not do this in production.

`latest` is a moving target. Today it might point to Node 20. Tomorrow, it might point to Node 22, severely breaking our app. We must strictly pin our versions (e.g., node:20.12.0-alpine) to ensure our code always works.

Cleaning Up the Mess

Over months of development, our hard drive will fill up with gigabytes of "dangling" images-old layers that are no longer associated with any active image tag because we rebuilt our code 500 times.

To safely nuke all unused, orphaned image layers and reclaim our disk space, we run the ultimate janitor command: 

docker image prune

If we want to aggressively wipe out absolutely every image not currently attached to a running container, we add the -a flag (docker image prune -a). This gives us a blank slate!
