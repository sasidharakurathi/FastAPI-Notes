The Problem: Containers are Ephemeral

By default, everything created inside a container is strictly temporary. If our app writes a log file or a user uploads an image, it is saved on the container's writable "top layer". When we type docker rm, the entire container is permanently deleted along with all its data.

We need a way to outlive the specific running container, especially for databases. This is where Volumes come in.

The Three Types of Mounts

Type What it is When we use it
Named Volumes Docker creates a fully managed folder hidden deep inside our host's OS. Docker exclusively handles it. Databases, stateful data, or when we don't care where the data is stored physically, we just want it saved safely.
Bind Mounts We actively attach a specific folder on our laptop (e.g., `/Users/dev/my_code`) into the container. Local development! We want real-time syncing so when we edit code in VS Code, the container instantly sees it.
tmpfs Mounts Data is stored directly in RAM. It never touches our physical hard drive. High-security applications where we never want sensitive API secrets to touch disk.

Volume Management Commands

Because Named Volumes are treated as first-class objects in Docker, we manage them directly via the CLI:

docker volume create pg_data
Creates an empty named volume ready to be attached.

docker volume ls
Lists all the volumes Docker is currently keeping alive on our machine.

docker volume inspect pg_data
Tells us the exact raw file path where our data is actually living on the host.

docker volume rm pg_data
Permanently deletes the volume. (Only works if no containers are currently attached to it).

The Great Debate: -v vs --mount

When running a container, we have two ways to attach a volume. Historically we used -v, but Docker now strongly recommends --mount because it is significantly clearer to read.

# The old, short way (-v)
docker run -v my_volume:/app/data my_app

# The modern, explicit way (--mount)
docker run --mount type=volume,source=my_volume,target=/app/data my_app
Why use --mount?
If we accidentally misspell our volume name using -v, Docker quietly creates a brand new, empty, unused volume on our host disk automatically. It doesn't tell us, leading to confusing blank databases. If we do that with --mount, Docker will instantly throw an error and stop safely.

Real Example 1: PostgreSQL with a Named Volume

If we try to run a database without a volume, the second the container stops, our entire database vanishes. Let's fix that by attaching a Named Volume so our user table survives a restart.

# 1. Create the persistent volume explicitly
docker volume create my_db_data

# 2. Run Postgres and map the volume to its expected data directory
docker run -d \
  --name my_postgres \
  -e POSTGRES_PASSWORD=secret \
  --mount type=volume,source=my_db_data,target=/var/lib/postgresql/data \
  postgres:15-alpine

Now, even if we run docker stop my_postgres && docker rm my_postgres, the moment we spin it back up using the exact same volume mount command, all our tables and rows will still be there waiting for us to use!

Real Example 2: Live Code Reload (Dev Workflow)

When developing locally, we don't want to wait 60 seconds to rebuild a Docker image just because we changed a single <h1> tag in our code. We use a Bind Mount to transparently project our live code folder directly into the container.

# We are inside our project folder where our 'server.js' lives.
# We mount our current directory ($PWD) directly into the image's /app folder.

docker run -d \
  --name dev_server \
  -p 3000:3000 \
  --mount type=bind,source="$(pwd)",target=/app \
  my_node_app npm run dev
⚠️ Warning: The node_modules Overwrite
If we bind mount our entire laptop's folder into a Node.js container, our local MacOS/Windows node_modules folder will violently overwrite the Linux node_modules folder compiled inside the container, likely breaking native C++ bindings (like bcrypt or sass).

Summary: When to Use Which Type?

To wrap up this chapter, here is the golden rule for choosing volume types:

  • Use Named Volumes for Production Data: If we are capturing database state, cache files, or media uploads, we want Docker to securely manage them. Named volumes are easier to back up, share among containers, and they guarantee high Linux filesystem performance.
  • Use Bind Mounts for Local Development: If we want our IDE (like VS Code) edits to be immediately reflected inside the running container without rebuilding the image, we must use a bind mount. Never use a bind mount in production.
  • Use tmpfs for In-Memory Secrets: If our application generates highly sensitive temporary keys, passwords, or cryptographic tokens, we write them to a `tmpfs` mount so no attacker can ever recover them from our hard drives after the container dies.
