CHAPTER 9 — Docker Compose (Multi-Container Apps)

What is Compose and Why Does it Exist?

Until now, we have been using docker run to start a single container. But real applications are rarely single containers. A modern web app might require a reverse proxy, a Python backend, a Redis cache, and a PostgreSQL database.

Typing out four massive docker run commands with 15 flags each, manually creating virtual networks, and hoping we do it in the exact right order is physically exhausting and highly error-prone.

Enter Docker Compose:
Compose is an IaC (Infrastructure as Code) tool. Instead of typing commands into the terminal, we write down our entire architectural layout in a single, human-readable YAML file called docker-compose.yml. Docker reads this file and magically spins up the networks, volumes, and containers in one shot.

The docker-compose.yml Structure

A Compose file is strictly divided into top-level blocks:

  • services: The actual containers we want to run (e.g., `web`, `db`).
  • volumes: The named volumes we want Docker to create and manage.
  • networks: The custom virtual networks our containers will communicate across.

The Everyday Compose Commands

Command What it does
docker compose up -d Reads the YAML, builds missing images, creates networks, and boots the entire stack in the background.
docker compose down -v Stops every container in the stack, deletes them, and deletes the custom networks. The -v flag completely wipes the attached volumes (useful for nuking the dev database).
docker compose build Forces Docker to re-read our Dockerfiles and rebuild the custom images without starting the stack.
docker compose logs -f Streams the combined stdout logs of all containers in the stack simultaneously.
docker compose exec db bash Drops us into an interactive bash shell inside the specific service named `db`.

Real Example 1: Full Stack Python App

Let's map out a complex, real-world application. We have a FastAPI backend, a Postgres database, a Redis cache, and an Nginx reverse proxy.

# docker-compose.yml
services:
  db:
    image: postgres:15-alpine
    environment:
      POSTGRES_PASSWORD: ${DB_PASSWORD} # Pulls from our hidden .env file
    volumes:
      - pg_data:/var/lib/postgresql/data
    restart: unless-stopped

  redis:
    image: redis:7-alpine
    restart: unless-stopped

  api:
    build: 
      context: ./backend
      dockerfile: Dockerfile
    command: uvicorn main:app --host 0.0.0.0 --port 8000
    depends_on:
      - db
      - redis
    environment:
      DATABASE_URL: postgresql://postgres:${DB_PASSWORD}@db:5432/postgres
      REDIS_URL: redis://redis:6379/0
    restart: unless-stopped

  proxy:
    image: nginx:alpine
    ports:
      - "80:80"   # The ONLY port we expose to our laptop/world
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
    depends_on:
      - api

volumes:
  pg_data: # Declares the named volume so Compose creates it
⚠️ Automatic Networking Magic:
Notice that we never declared a networks: block in that file. That is because Docker Compose automatically creates a dedicated bridge network for the stack, and permanently attaches every service to it. They can instantly talk to each other using their service names (`db`, `redis`, `api`).

Real Example 2: Node.js + MongoDB

Let's look at another stack. If we are developing a Node application, we might want to map our laptop's folder directly into the container for live reloading.

services:
  mongo:
    image: mongo:6
    volumes:
      - mongo_data:/data/db

  node_app:
    build: .
    ports:
      - "3000:3000"
    volumes:
      # Bind mount our laptop directory to live-reload code
      - ./:/app
      # Anonymous volume trick: Prevents our local node_modules
      # from overwriting the container's built linux node_modules
      - /app/node_modules
    env_file:
      - .env
    command: npm run dev

volumes:
  mongo_data:

Environment Variable Management

Writing highly sensitive passwords directly in a docker-compose.yml file is a massive security failure. If we commit that to Git, our servers will absolutely get hacked.

Instead, we use .env files. Docker Compose automatically searches for a `.env` file in the same directory as the `docker-compose.yml`. We can dynamically inject those secrets in two ways:

  1. Variable Substitution: We write ${MY_SECRET} directly in the YAML file (like we did with Postgres).
  2. The env_file Key: We pass the entire `.env` file wholesale straight into the container environment (like we did with our Node app).

Advanced: Override Files (Dev vs. Prod)

Our infrastructure behaves differently on our laptop than it does in production. We don't want two completely separate Compose files, because they will quickly drift out of sync.

Docker naturally supports Override Files. If we have a file named docker-compose.override.yml, Docker heavily merges it with the base docker-compose.yml file when we type `docker compose up`.

  • Base (`docker-compose.yml`): Holds the generic setup (images, basic networks). Committed to Git.
  • Override (`docker-compose.override.yml`): Contains our local dev overrides (massive bind mounts for live reloading, exposing internal database ports to our laptop for debugging). We add this to .gitignore!

The depends_on Startup Illusion

The depends_on field tells Compose to start the Database before it starts the Python API.

The Startup Paradox:
depends_on solely guarantees that Compose will send the start command to the DB container first. However, a database might take 15 seconds to allocate memory and accept connections. Compose does NOT wait. It immediately starts the Python API right after, which promptly crashes because the DB is booting but not actually listening yet.

We solve this tightly coupling using actual Healthchecks. We can define exactly how to test if the DB is ready, and explicitly tell our API to wait for that passing grade.

services:
  db:
    image: postgres:15
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      retries: 5

  api:
    image: my_api
    depends_on:
      db:
        condition: service_healthy # The API will pause and wait!
